## 1. Backend — Alterações no Schema Prisma

### 1.1 Model `User` — Novos campos globais

```prisma
model User {
  // ... campos existentes ...
  cpf         String?  @unique
  country     String?
  state       String?
  city        String?
  neighborhood String?
  street      String?
  number      String?
  complement  String?
}
```

Justificativa: CPF e endereço são dados da pessoa, não da igreja (Opção A aprovada pelo usuário).

### 1.2 Model `Branch` — Endereço estruturado

```prisma
model Branch {
  // ... campos existentes ...
  // Remover: address String?
  country      String?
  state        String?
  city         String?
  neighborhood String?
  street       String?
  number       String?
}
```

Justificativa: Design da tela 10 exige campos separados.

### 1.3 Endpoints impactados

| Endpoint | Mudança |
|----------|---------|
| `GET /auth/me` | Response já inclui User — novos campos vêm automaticamente |
| `GET /member-profiles/:id` | Incluir `user.cpf`, `user.country`, etc. no `include` |
| `GET /member-profiles` | Idem para listagem |
| `PATCH /auth/me` (NOVO) | Permitir usuário editar próprio perfil (nome, cpf, endereço, phone) |
| `GET /branches` | Adicionar `_count.memberships` no response |
| `POST /branches` | Aceitar novos campos de endereço |
| `PATCH /branches/:id` | Aceitar novos campos de endereço |
| `PATCH /memberships/:id` | Já suporta mudar `status` para INACTIVE (tela 13) |

### 1.4 Novo Endpoint

- `PATCH /auth/me` — Atualiza dados do próprio usuário (nome, cpf, phone, email, endereço). Apenas o próprio usuário autenticado. Retorna user atualizado.

## 2. Mobile — Novas Telas e Componentes

### 2.1 Tela 06 — Perfil do Membro (`MemberProfilePage`)

- **Rota:** `/member-profile/:id`
- **Feature:** `features/profile/`
- **Cubit:** `MemberProfileCubit` — carrega dados do membro via `/member-profiles/:id`
- **UI:** Avatar circular, nome, tags de posições (chips), datas (batismo, membro desde), seção dados pessoais (telefone, email, endereço), ações da secretaria (Carta de Recomendação, Certificado EBD — ambos "Em breve")
- **Botão:** "Editar Perfil" navega para tela 07 ou 11 dependendo do contexto (perfil próprio vs membro)

### 2.2 Tela 07 — Editar Perfil (`EditProfilePage`)

- **Rota:** `/edit-profile`
- **Feature:** `features/profile/`
- **Cubit:** `EditProfileCubit` — carrega dados via `/auth/me`, salva via `PATCH /auth/me`
- **UI:** Formulário com foto (tap para alterar), campos: nome completo, CPF, telefone, email, país, estado, cidade, bairro, rua, número, complemento. Botão "Salvar".
- **Validação:** CPF formato válido, email formato válido, campos obrigatórios (nome)

### 2.3 Tela 08 — Drawer (`AppDrawer`)

- **Tipo:** Widget reutilizável
- **Local:** `shared/widgets/app_drawer.dart`
- **UI:** Header com logo + nome da igreja + "Church Management". Seção "Módulos Core": Secretaria, Finanças, Patrimônio, Congregações. Seção "Ferramentas": Relatórios, Certificados, Contribuições, Célula. Footer: System Status.
- **Navegação:** Secretaria → `MembersPage` (existente), Congregações → tela 12. Demais → "Em breve".
- **Integração:** Substituir bottom nav atual. Novo bottom nav: Início, Agenda ("Em breve"), Notificações ("Em breve").

### 2.4 Tela 09 — Bottom Sheet de Perfil (`ProfileBottomSheet`)

- **Tipo:** Modal Bottom Sheet
- **Local:** `shared/widgets/profile_bottom_sheet.dart`
- **Acionamento:** Tap no avatar do usuário no dashboard
- **UI:** Avatar + nome + cargo ("Gestor da Comunidade"), opções: Meu Perfil (→ tela 06 com próprio ID), Configurações ("Em breve"), Sair (logout). Versão do app no footer.

### 2.5 Tela 10 — Cadastrar Congregação (`CreateBranchPage`)

- **Rota:** `/create-branch`
- **Feature:** `features/branches/`
- **Cubit:** `BranchFormCubit` — `POST /branches`
- **UI:** Formulário com ícone + título "Cadastrar Congregação", seção "Dados da Unidade" (nome, tipo — derivado de isHeadquarters), seção "Localização" (país, estado, cidade, bairro, rua, número). Botões: Cancelar, Salvar Unidade.

### 2.6 Tela 11 — Editar Membro (`EditMemberPage`)

- **Rota:** `/edit-member/:id`
- **Feature:** `features/members/`
- **Cubit:** `EditMemberCubit` — carrega via `/member-profiles/:id`, salva via `PATCH /member-profiles/:id` + `PATCH` user data
- **UI:** Avatar + nome, seção "Informações Pessoais" (nome, email, CPF, telefone, data nascimento), seção "Dados Eclesiásticos" (data batismo, data admissão, número de registro, congregação), seção "Zona de Perigo" (botão "Inativar Membro" → tela 13). Botão "Salvar Registro".

### 2.7 Tela 12 — Lista de Congregações (`BranchesPage`)

- **Rota:** `/branches`
- **Feature:** `features/branches/`
- **Cubit:** `BranchesCubit` — `GET /branches` (com contagem de membros)
- **UI:** Título "Congregações" + subtítulo, campo de busca, lista de cards (nome, badge de membros na sede, endereço, link "Ver Detalhes"), FAB "+" para criar nova (→ tela 10).

### 2.8 Tela 13 — Confirmação de Inativação (`InactivateMemberDialog`)

- **Tipo:** AlertDialog
- **Local:** `features/members/presentation/widgets/inactivate_member_dialog.dart`
- **UI:** Título "Inativar Membro?", texto explicativo, botões: Cancelar (outline), Inativar (vermelho).
- **Ação:** `PATCH /memberships/:id` com `status: INACTIVE`

## 4. Abordagem TDD

Seguindo o padrão do projeto (memory `feedback_testing_strategy`):

### Backend (Jest)
- Testes de service para cada método novo/alterado
- Testes de controller para validação de DTOs e guards
- Mocks do Prisma via `jest.mock`
- Nomenclatura: `describe('ServiceName - methodName')` → `it('should ...')`

### Mobile — Cubits (bloc_test + mocktail)
- Cada cubit novo: testes de estados emitidos (loading → loaded/error)
- Edge cases: dados vazios, erro de rede
- Nomenclatura: `group('CubitName - action')` → `test('should ...')`

### Mobile — Widgets (flutter_test)
- Renderização dos elementos visuais
- Interações (tap, submit, navegação)
- Estados de loading/error/empty
- Nomenclatura: `group('WidgetName')` → `testWidgets('should ...')`

### Verificação Final
- Backend: `npx nest build` ou `npx tsc --noEmit`
- Mobile: `flutter analyze`
